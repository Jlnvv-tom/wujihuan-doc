# 第8章 从0到1全流程产品设计

学了很多方法论，但真要你从0到1做一个产品，还是不知道从哪下手？问题不是你学得不够，而是没有人带你走一遍完整流程。

我是怕浪猫，这一章用一个完整的案例，带你走一遍"从点子到产品"的全流程。不是每个环节的简单罗列，而是真实的决策过程和取舍思考。

## 8.1 从点子到产品

### 8.1.1 产品决策方法论

产品决策的核心问题是："做还是不做？"

很多产品经理面对新想法时，要么"老板说做就做"，要么"我觉得值得做就做"。这两种决策方式都不可靠。可靠的决策方法是"假设验证法"。

**假设验证法的三步操作**

第一步：把想法转化为假设

不要说"我们要做一个社区功能"，要说"假设上线社区功能后，用户7日留存率会提升5%"。假设必须是可验证的。

第二步：设计验证方案

怎么验证这个假设？不是直接开发社区功能，而是用最小成本验证。可能的方案：

- 用户调研：问100个用户，他们是否需要社区功能
- 数据分析：看现有用户的行为数据，有没有自发形成的社区行为
- 灰度测试：上线最简单的社区MVP，看数据表现

第三步：根据验证结果决策

验证结果支持假设 → 做这个功能
验证结果否定假设 → 不做或调整假设
验证结果模糊 → 缩小范围继续验证

> 产品决策不是"我觉得"，而是"假设→验证→决策"。每一次产品决策背后，都应该有一个可验证的假设。

**假设验证的常见错误**

假设验证说起来简单，但执行中踩坑的人太多了。怕浪猫见过太多团队，明明做了验证，结果还是翻车。问题出在哪？出在验证本身就有偏差。

错误一：验证偏差

你设计验证方案的时候，方案本身就偏向你想要的结果。比如你想验证"用户需要AI写作功能"，你去写作社区发问卷——这些人本来就是重度写作用户，当然会说需要。你只是在验证"重度写作用户需要AI写作"，而不是"普通用户需要AI写作"。

典型表现：只在核心用户群里做调研，只在 favorable 的场景下测试，只收集支持假设的数据。验证偏差的本质是"你太想证明自己是对的"，所以不知不觉地设计了有利于自己的实验。

怎么避免？找一组"反对者"——找那些你觉得不会用这个功能的人，看他们的反应。如果连反对者都说不做不行，那假设才算真正被验证了。

错误二：幸存者偏差

你只看到了"留下来的人"的数据，没看到"已经走掉的人"的数据。比如你做了一版新功能，上线后看到使用这个功能的用户留存率很高，就得出结论"新功能提升了留存"。但你没看到那些因为新功能太复杂而直接卸载的用户——他们已经不在你的数据里了。

典型表现：只分析活跃用户的行为数据，只收集留下来的用户的反馈，忽略了流失用户的沉默信号。幸存者偏差在产品决策中极其隐蔽，因为你"看不到你没看到的东西"。

怎么避免？每次做验证的时候，强制自己回答一个问题："那些没参与验证的人，他们为什么不参与？"追踪流失用户、分析沉默数据、看卸载原因，这些"反面数据"比正面数据更有价值。

错误三：确认偏差

人天生倾向于寻找支持自己观点的证据，忽略反对的证据。这不是故意的，是大脑的默认设置。产品经理更是重灾区——你的idea，你做的方案，你当然希望它成功。于是你会在验证过程中不自觉地放大正面信号、忽略负面信号。

典型表现：看到3个正面反馈就当结论，看到3个负面反馈就当个案；问卷设计的问题本身就暗示了期望答案；解读数据时选择性地强调有利指标。

怎么避免？两招。第一，验证方案设计完之后，找一个"唱反调"的同事来审你的方案，看他能不能指出你的偏向。第二，预先定义"什么结果算否定"——在验证之前就写下来，如果留存率提升不到2%，就算假设不成立。别等结果出来再调整标准。

| 偏差类型 | 典型表现 | 核心风险 | 避免方法 |
|----------|---------|---------|---------|
| 验证偏差 | 只在有利场景下验证 | 验证结果不代表真实情况 | 找反对者做反向验证 |
| 幸存者偏差 | 只看留下来的人 | 忽略流失用户的信号 | 追踪沉默数据，分析流失原因 |
| 确认偏差 | 选择性接受正面证据 | 自我说服，忽视风险 | 预设否定标准，找人唱反调 |

**如何设计最小化验证实验**

假设验证的关键不是"验证"，而是"最小化"。用最小的成本、最短的时间，得到一个可信的结论。怕浪猫给你一套操作模板。

第一步：定义可衡量的验证目标

模糊的目标没法验证。"用户喜欢"不是目标，"40%的测试用户会在3天内回访"才是目标。验证目标必须包含三个要素：指标、阈值、时间窗口。

| 要素 | 说明 | 示例 |
|------|------|------|
| 指标 | 你要测量什么 | 7日留存率 |
| 阈值 | 达到多少算验证通过 | 提升5个百分点 |
| 时间窗口 | 多久内测量 | 上线后14天 |

第二步：选择最小化的验证方式

不是所有验证都需要写代码。按照成本从低到高，验证方式可以这样排：

| 验证方式 | 成本 | 周期 | 适用场景 |
|----------|------|------|---------|
| 桌面研究 | 极低 | 1-2天 | 验证市场是否存在、竞品分析 |
| 用户访谈 | 低 | 3-5天 | 验证用户痛点是否真实 |
| 问卷调研 | 低 | 3-7天 | 验证需求覆盖面和优先级 |
| 落地页测试 | 中 | 1-2周 | 验证用户是否愿意"掏钱" |
| 竞品数据爬取 | 中 | 1-2周 | 验证市场天花板 |
| MVP测试 | 高 | 2-4周 | 验证核心假设和关键行为 |

关键原则：能用低成本方式验证的，不要用高成本方式。能用问卷搞定的，不要开发MVP。

第三步：控制变量，隔离干扰

验证实验最忌讳"多个变量一起变"。你同时上线了三个新功能，数据涨了，你说是哪个功能带来的？说不清。所以每次验证只改一个变量，其他条件保持不变。

第四步：设定判断标准，不要事后调整

验证之前就写好：什么结果算通过，什么结果算不通过。不要等数据出来了再说"这个指标不太好看，我们换一个指标"。这和考试完了改答案一样，没有意义。

举个例子：你要验证"AI推荐能提升购买转化率"。预设标准是"实验组购买转化率比对照组提升3个百分点，p值<0.05"。跑完实验，结果提升了1.8个百分点，p值0.12。结论就是"假设未验证通过"，而不是"趋势是正向的，再跑一周看看"。

> 最小化验证的核心是"用最小成本得到可信结论"。不是"用最小成本得到你想听的结论"。如果你发现自己一直在调标准，说明你不是在做验证，你是在自我安慰。

### 8.1.2 需求筛选与方案取舍

不是所有的需求都值得做。需求筛选的核心框架是"价值×可行×紧迫"。

**需求筛选矩阵**

| 维度 | 评估方法 | 权重 |
|------|---------|------|
| 用户价值 | 解决多少用户的核心问题 | 40% |
| 商业价值 | 对关键指标的影响 | 30% |
| 技术可行性 | 实现难度和风险 | 20% |
| 紧迫性 | 时间窗口和竞争态势 | 10% |

**方案取舍的三个原则**

原则一：优先做"ROI最高的"。不是"最重要的"，也不是"最紧急的"，而是"投入产出比最高的"。

原则二：砍掉"不确定的"。如果对某个需求的效果不确定，优先做验证而不是直接开发。

原则三：推迟"不紧急的"。每个版本只做最重要的事，不紧急的放进下一版。

### 8.1.3 MVP 定义与范围裁剪

MVP（Minimum Viable Product）是最小可行产品。不是"最简单的产品"，而是"能验证核心假设的最小产品"。

**MVP定义三步法**

第一步：定义核心假设。你的产品最核心的假设是什么？

举例：一个AI写作工具的核心假设是"用户愿意用AI辅助写作"。

第二步：识别验证路径。验证这个假设最少需要哪些功能？

AI写作工具验证"用户愿意用AI辅助写作"的最小功能：输入框+AI生成按钮+结果展示。不需要登录注册、不需要付费系统、不需要个人中心。

第三步：裁剪到最小。把功能裁剪到"只保留验证核心假设所需的最小集合"。

**MVP裁剪对比表**

| 功能 | MVP版本 | V1版本 | V2版本 |
|------|---------|--------|--------|
| AI生成 | 有 | 有 | 有 |
| 登录注册 | 无（游客模式） | 有 | 有 |
| 付费系统 | 无 | 无 | 有 |
| 历史记录 | 无 | 有 | 有 |
| 模板市场 | 无 | 无 | 有 |
| 分享功能 | 无 | 无 | 有 |

> MVP不是"简陋的半成品"，而是"精准的试验品"。每一行裁剪都是在说"这个功能对验证核心假设没有帮助，先不做"。

**MVP的三种形态**

MVP不是只有一种样子。根据你要验证的假设类型不同，MVP的形态也不同。怕浪猫见过太多人把"功能精简版"当成唯一的MVP形态，结果验证成本居高不下。其实选对了MVP形态，验证成本可以降一个数量级。

形态一：冒烟测试MVP（Smoke Test MVP）

核心逻辑：先做一个"看起来像真的"的页面或广告，看用户有没有兴趣，再决定要不要做真的。本质上是验证"用户有没有这个需求"，而不是"产品能不能满足需求"。

操作方法：做一个落地页，描述你的产品功能和价值，放一个"立即体验"或"预约"按钮。用户点击后，你告诉他"产品还在开发中，留下邮箱第一时间通知你"。你不需要写一行真正的产品代码，只需要一个网页。

适用场景：验证需求是否存在、验证用户是否愿意"掏钱"。特别适合全新品类——没人做过，你不确定市场存不存在。

数据指标：落地页的访问→点击转化率、邮箱留资率。行业参考值：访问→点击转化率>15%算有需求，留资率>5%算兴趣较强。

真实案例：Dropbox最早的MVP就是一个3分钟的视频，演示了文件同步的功能。视频放出后，等待列表从5000人涨到了75000人。一个视频，验证了"用户需要跨设备文件同步"这个假设，成本接近零。

形态二：Wizard of Oz MVP（绿野仙踪MVP）

核心逻辑：前端看起来是自动化的，后端其实是人工操作的。用户以为在和AI对话，其实后面坐着一个真人。用户以为系统在自动匹配，其实是运营在手动分配。

操作方法：搭建一个前端界面，让用户体验完整的流程。但核心功能由人工在后台完成。用户的请求进来，你手动处理，然后把结果返回给用户。

适用场景：验证用户是否愿意为这个服务买单、验证核心体验是否有价值。特别适合AI产品和服务型产品——算法还没ready，但你想先验证用户需不需要这个服务。

数据指标：用户使用频次、付费意愿、NPS。关键不是看"自动化能不能实现"，而是看"用户愿不愿意用这个服务"。

真实案例：Zappos（后来被亚马逊收购）最早验证"用户愿意在网上买鞋"的假设时，没有进任何货。创始人去鞋店拍照片放到网站上，有人下单了，他就去鞋店买一双寄过去。全部人工操作，零库存，验证了"用户愿意在网上买鞋"这个核心假设。

形态三：功能精简MVP（Concierge MVP的进阶版）

核心逻辑：只开发验证核心假设所需的最少功能，其他功能全部砍掉。这就是最常见的MVP形态——前面讲的"三步法"产出的就是这种。

操作方法：按照前面讲的MVP定义三步法，定义核心假设→识别验证路径→裁剪到最小。核心是"每一行裁剪都要能说出理由"。

适用场景：核心假设涉及具体的产品体验，必须让用户真正用过才能判断。比如"用户会不会用语音交互代替打字"，这个假设光靠落地页和人工模拟都不够，必须让用户真的说一句话试试。

数据指标：核心行为完成率、留存率、回访率。和预设的验证阈值对比，决定假设是否通过。

**三种MVP形态对比**

| 形态 | 验证什么 | 开发成本 | 验证周期 | 可扩展性 | 典型场景 |
|------|---------|---------|---------|---------|---------|
| 冒烟测试MVP | 需求是否存在 | 极低（1个落地页） | 1-3天 | 低 | 全新品类、市场验证 |
| Wizard of Oz MVP | 体验是否有价值 | 中（前端界面+人工后端） | 1-2周 | 低（人工瓶颈） | AI产品、服务型产品 |
| 功能精简MVP | 产品能否被使用 | 高（最少可用功能） | 2-4周 | 高 | 需要真实体验验证 |

选型的关键判断：你要验证的是"有没有需求"，选冒烟测试；你要验证的是"体验好不好"，选Wizard of Oz；你要验证的是"功能行不行"，选功能精简。别上来就选最重的那个，先用最轻的方式探路，探不明白再加重。

## 8.2 从决策到落地

### 8.2.1 清晰表达产品思路

产品思路的表达，我推荐"电梯演讲法"：在30秒内把你的产品思路讲清楚。

**电梯演讲模板**

我们的产品是[产品名称]，帮助[目标用户]解决[核心问题]，通过[核心方案]，让用户获得[核心价值]。和[主要竞品]相比，我们的差异化是[差异化点]。

举例：

"我们的产品是'笔灵AI'，帮助自媒体创作者解决'写稿效率低'的问题，通过AI辅助写作，让用户在3分钟内完成一篇高质量草稿。和ChatGPT相比，我们针对中文内容创作做了专项优化，并提供了行业模板和素材库。"

30秒，核心信息全部传达。如果你的产品思路说不清，说明你自己还没想清楚。

### 8.2.2 画出完整原型

有了产品思路后，接下来就是画原型。

**原型画法的核心原则**

第一，先画主流程。从用户打开App到完成核心操作的全路径。不要一开始就画所有页面的细节。

第二，标注关键交互。每个页面的核心交互是什么？点击后跳转到哪里？有什么状态变化？

第三，标注数据字段。列表页需要展示哪些字段？详情页需要展示哪些信息？这些在原型上用注释标注。

第四，覆盖异常场景。空状态、错误状态、加载状态、边界情况——每一页都要画。

**原型评审前自检清单**

| 检查项 | 是否完成 |
|--------|---------|
| 主流程的页面是否齐全 |
| 每个页面的交互是否标注 |
| 异常流程是否覆盖 |
| 数据字段是否标注 |
| 状态变化是否说明 |
| 页面之间的跳转是否连贯 |

### 8.2.3 写出规范 PRD

PRD的规范写法在第7章已经详细讲过。这里补充几个"从0到1"场景下的特殊注意事项：

**注意一：背景部分要讲清楚"为什么做"**

从0到1的产品，团队对你做的事情可能完全不了解。PRD的背景部分要比迭代项目更详细——市场机会是什么、用户痛点是什么、为什么是现在做。

**注意二：功能范围要明确"做什么"和"不做什么"**

从0到1的产品最容易犯的错误是"功能蔓延"。PRD中要明确标注"本期不做的功能"和"后续迭代考虑的功能"，避免评审时被带偏。

**注意三：技术方案要和开发提前对齐**

从0到1的产品涉及技术选型，PRD中的技术约束部分要和开发负责人提前沟通，确保方案可行。

### 8.2.4 从0到1产品设计实战案例：社区团购小程序

讲了这么多方法论，怕浪猫知道你一定在想：能不能来个完整的？行，这一节带你走一遍真实的从0到1。以"社区团购小程序"为例，从需求发现到全流程串通，每一步的决策和取舍都给你拆开看。

**第一步：需求发现**

场景：你所在的公司主营生鲜供应链，服务B端商户。老板提出要做C端业务，方向是社区团购。你作为产品经理，需要从0开始。

需求发现不是"老板说什么就做什么"，而是要搞清楚三件事：用户是谁、痛点是什么、市场空间有多大。

你做了这些事情：

桌面研究：社区团购市场2023年规模超过3000亿，头部玩家（多多买菜、美团优选）占据70%份额，但下沉市场仍有空间。头部平台的问题是什么？品质不稳定、配送不及时、售后体验差。

用户访谈：你找了20个社区团购用户深聊，核心发现是：用户对价格敏感但不盲目低价，最在意的是"品质稳定"和"取货方便"。小区团长是关键节点——用户因为信任团长才下单，不是因为信任平台。

竞品分析：头部平台标准化程度高，但灵活度低。团长工具重、操作复杂，小团长用不起来。下沉市场的团长普遍年龄偏大，对复杂操作有抵触。

需求发现阶段的结论：市场存在，但正面硬刚头部平台没有胜算。差异化切入口是"让团长用得更简单"，用极简的团长工具撬动下沉市场。

**第二步：假设验证**

你定义了三个核心假设，逐一验证。

假设1：下沉市场的团长对现有工具不满，愿意换更简单的工具。

验证方式：用户访谈+问卷。你联系了50个下沉市场的团长（通过供应链渠道触达），45个表示现有工具"太复杂了"，38个表示"如果有更简单的愿意试试"。

验证结果：假设成立。团长对工具简洁性有强需求。

假设2：团长通过微信社群可以带动C端用户下单。

验证方式：冒烟测试MVP。你做了一个落地页，描述"社区团购，团长一键开团，居民群里接龙下单"，放了一个"我想当团长"的按钮。在3个小区的业主群里投放。

验证结果：3天收到27个团长申请，落地页点击→申请转化率18%。远超5%的参考线。

假设3：C端用户愿意通过小程序在团长群里买生鲜。

验证方式：Wizard of Oz MVP。你做了一个前端小程序（只有商品展示和下单按钮），但后端全是人工处理——用户下单后，你在后台手动记录，手动通知供应链发货，手动跟进配送。

验证结果：跑了2周，日均订单43单，复购率38%。用户不在乎"系统简陋"，在乎的是"菜新不新鲜、取货方不方便"。

假设验证汇总表：

| 假设 | 验证方式 | 样本量 | 关键指标 | 结果 |
|------|---------|--------|---------|------|
| 团长对现有工具不满 | 访谈+问卷 | 50人 | 76%表示不满，愿意换 | 通过 |
| 团长愿意入驻新平台 | 冒烟测试 | 3个小区投放 | 点击→申请转化率18% | 通过 |
| C端用户愿意在小程序下单 | Wizard of Oz | 2周运营 | 日均43单，复购率38% | 通过 |

三个假设全部通过，可以进入MVP定义阶段。如果任何一个假设未通过，都要回到需求发现阶段重新审视。

**第三步：MVP定义**

核心假设已经验证通过，现在定义MVP的范围。

MVP的核心假设：团长用极简工具就能完成开团、跟单、提货全流程，且愿意持续使用。

MVP验证路径：团长端需要"开团→分享→跟单→提货确认"的最小闭环，C端需要"浏览→下单→取货"的最小闭环。

MVP功能裁剪：

| 功能模块 | MVP版本 | V1版本 | V2版本 | 裁剪理由 |
|----------|---------|--------|--------|---------|
| 团长开团 | 有 | 有 | 有 | 核心闭环必须 |
| 团长分享到群 | 有（生成海报+链接） | 有 | 有 | 核心闭环必须 |
| C端浏览商品 | 有 | 有 | 有 | 核心闭环必须 |
| C端下单 | 有 | 有 | 有 | 核心闭环必须 |
| 微信支付 | 有 | 有 | 有 | 核心闭环必须 |
| 团长跟单 | 有（简单列表） | 有（详情+筛选） | 有 | 精简为列表，详情后续补 |
| 提货确认 | 有（团长手动确认） | 有 | 有 | 核心闭环必须 |
| 售后退款 | 无（人工处理） | 有 | 有 | MVP阶段人工处理即可 |
| 商品管理 | 无（后台配置） | 有 | 有 | MVP阶段运营代配置 |
| 配送跟踪 | 无 | 无 | 有 | 非核心假设，V2再做 |
| 营销活动 | 无 | 有 | 有 | 非核心假设 |
| 评价系统 | 无 | 有 | 有 | 非核心假设 |
| 数据报表 | 无 | 有（基础） | 有（完整） | MVP阶段不需要 |
| 多角色权限 | 无 | 无 | 有 | 团长就一个人，不需要权限 |

MVP的交付标准：团长能在5分钟内完成开团并分享到群；C端用户能在3步内完成下单；团长能在1个页面内查看所有待取货订单。

**第四步：功能设计**

MVP的功能设计聚焦"极简"二字。怕浪猫带你过团长端和C端的核心页面。

团长端核心流程（4步闭环）：

选择商品（从运营配置好的商品列表中勾选）→ 一键开团（生成团购链接和海报）→ 分享到群（点击分享按钮，直接跳转微信分享）→ 跟单提货（一个列表页，显示所有待取货订单，点击确认提货）

C端核心流程（3步闭环）：

浏览商品（从团长分享的链接进入，看到本次团购的商品列表）→ 选择下单（选商品、选数量、填取货信息）→ 支付完成（微信支付，跳转订单详情页）

每个核心页面的异常场景：

团长开团页：没有可选商品（展示"暂无商品，联系运营"）、网络失败（展示重试提示）

C端浏览页：团购已结束（展示"本次团购已结束，关注团长获取下次通知"）、商品售罄（展示"已售罄"标记，不可点击）

C端下单页：库存不足（选择数量时实时校验，超出库存提示"最多可购买X份"）

支付页：支付失败（提示重新支付，订单保留10分钟）

团长跟单页：订单为空（展示"暂无订单"的空状态提示）

**第五步：全流程串通**

画完原型、写完PRD，做一遍全流程串通。怕浪猫模拟一个用户的完整路径：

场景：团长王姐在小区业主群里分享了团购链接，居民李阿姨看到后下单买水果。

1. 李阿姨点击群里的团购链接 → 进入C端商品列表页。页面展示：团购标题、截止时间、商品卡片（图片+名称+价格+库存）。
2. 李阿姨看到山东苹果5斤29.9元 → 点击"加入购物车" → 数量选择器弹出，选了1份 → 底部出现"去结算"按钮，显示金额29.9元。
3. 点击"去结算" → 进入确认订单页。展示：商品信息、取货地点（默认团长地址）、取货时间、备注。点击"提交订单"。
4. 进入微信支付 → 支付成功 → 跳转订单详情页。展示：订单号、商品信息、取货地点、取货时间、团长联系方式。
5. 第二天，水果到货。团长王姐在小程序里看到待取货订单列表，找到李阿姨的订单 → 点击"确认提货" → 订单状态变为"已提货"。
6. 李阿姨再次打开小程序 → 看到订单状态已变为"已提货"。

走完这个流程，你发现了什么问题？怕浪猫发现了这些：

问题1：李阿姨怎么知道水果到货了？MVP没有"到货通知"功能。需要加一个模板消息推送，否则李阿姨得自己来问团长。

问题2：如果李阿姨收到水果发现品质有问题怎么办？MVP没有售后入口。虽然售后人工处理，但至少要给用户一个"联系团长"的入口。

问题3：团长王姐的待取货订单如果很多，怎么快速找到特定人的订单？MVP的简单列表没有搜索功能。至少加一个按手机号后4位搜索的功能。

这些问题就是在全流程串通中发现的。不做串通，这些漏洞会一直藏到开发甚至上线之后。

### 8.2.5 全流程串通与自检

PRD写完后，不要立刻提交评审。先做一遍"全流程串通"。

**全流程串通的方法**

把自己当作用户，从打开产品开始，走完核心流程的每一步。每走到一个节点，问自己：

- 这个页面的目的是什么？
- 用户会点击哪里？
- 点击后会发生什么？
- 如果出了问题怎么办？
- 下一步要去哪里？

如果任何一个节点你回答不出来，说明PRD有漏洞。

**全流程串通检查表**

| 检查维度 | 检查内容 |
|----------|----------|
| 完整性 | 每个页面都有对应的设计说明吗？ |
| 一致性 | 同样的交互在不同页面表现一致吗？ |
| 连贯性 | 页面之间的跳转逻辑通顺吗？ |
| 容错性 | 异常情况都有处理方案吗？ |
| 可测性 | 每个功能都有验收标准吗？ |

> 从0到1最怕的不是"做错了"，而是"没想清楚就动手"。全流程串通一次，能发现PRD里90%的漏洞。

**全流程checklist：每个环节的交付物和验收标准**

从0到1的产品设计，每个环节都有明确的交付物和验收标准。怕浪猫给你整理一张完整的checklist，下次做项目直接照着打勾。

| 环节 | 交付物 | 验收标准 | 负责人 |
|------|--------|---------|--------|
| 需求发现 | 市场分析报告+用户画像 | 覆盖3个以上信息源，用户画像有访谈数据支撑 | 产品经理 |
| 假设验证 | 假设清单+验证方案+验证报告 | 每个假设有明确的验证方式、样本量、判断标准 | 产品经理 |
| MVP定义 | MVP功能清单+裁剪说明 | 核心假设有对应的验证路径，裁剪理由可追溯 | 产品经理 |
| 产品思路 | 电梯演讲文档 | 30秒内能讲清产品定位、用户、价值、差异化 | 产品经理 |
| 原型设计 | 完整原型+交互说明 | 主流程无断点，异常场景已覆盖，数据字段已标注 | 产品经理 |
| PRD撰写 | 完整PRD文档 | 功能范围明确（含"不做什么"），技术方案已对齐 | 产品经理 |
| 全流程串通 | 串通记录+问题清单 | 至少跑完3条核心用户路径，记录所有发现的问题 | 产品经理 |
| 评审 | 评审会议纪要+修改清单 | 所有问题有明确的处理方案和负责人 | 产品经理+团队 |

验收标准的核心逻辑：每个交付物都要能回答"怎么判断这个东西做完了"。如果验收标准写的是"完成调研"，这不是验收标准，这是任务描述。"完成10个用户深度访谈，输出3个用户画像"才是验收标准。

**流程卡点识别：哪些环节容易出问题**

从0到1的流程看起来很顺，但怕浪猫经历过太多项目，知道有些环节是天然的高危区。提前识别这些卡点，才能提前预防。

卡点一：假设验证环节——"验证做完了但没人信"

问题表现：验证做完了，但老板或团队不认可结论，觉得样本量不够、方式不对，要求"再验证一轮"。反复验证，项目停滞。

根本原因：验证之前没有和利益相关方对齐"验证标准和判断规则"。你觉得验证通过了，但决策者没参与标准的制定，自然不认。

预防方法：在验证方案设计阶段，就把"什么结果算通过、什么结果算不通过"写下来，和老板、技术负责人一起确认。验证结果出来后，按之前定的标准执行，不给"再看看"留空间。

卡点二：MVP定义环节——"越裁越大"

问题表现：MVP定义的时候，每个人都能说出"这个功能不加不行"的理由。一开始MVP只有5个功能，评审完变成12个。MVP变成了V1。

根本原因：没有清晰的核心假设作为裁剪锚点。每个功能看起来都很重要，因为没有统一标准来判断"重不重要"。

预防方法：MVP定义的第一步就是写清楚核心假设。所有功能都问一个问题："没有这个功能，核心假设还能验证吗？"能验证就砍。评审时如果有人要加功能，让他回答这个问题，答不上来就不加。

卡点三：PRD撰写环节——"技术方案反复推翻"

问题表现：PRD写完了，评审时技术说"这个方案做不了"，推翻重来。改完再评，又说"这个性能扛不住"，再推翻。反复几次，产品经理和开发的关系降到冰点。

根本原因：PRD写完才找开发对齐技术方案，太晚了。从0到1的项目技术不确定性高，等文档写完了再对齐，发现问题的成本很高。

预防方法：PRD撰写之前，先和开发负责人做一次技术预沟通。把核心流程和关键约束讲清楚，让开发评估可行性和风险点。技术预沟通不是让开发替你做设计，而是提前排除不可行的方案。

卡点四：全流程串通环节——"串通变成了走马观花"

问题表现：做了全流程串通，但只是"看了看页面"，没有真正模拟用户操作。串通完了觉得没问题，上线后问题一堆。

根本原因：串通时只看了"正常流程"，没有刻意测试边界场景和异常路径。人类大脑天然会走最优路径，但用户不会。

预防方法：串通时强制自己走3种角色路径（新用户、老用户、异常用户），每种角色至少走2条路径（主路径和1条异常路径）。准备一份"边界场景清单"，专门测试极端情况。

卡点识别汇总表：

| 卡点 | 环节 | 根本原因 | 预防方法 |
|------|------|---------|---------|
| 验证结论不被认可 | 假设验证 | 验证前未对齐判断标准 | 预设标准，利益相关方签字 |
| MVP越裁越大 | MVP定义 | 缺乏裁剪锚点 | 核心假设驱动，每个功能过筛 |
| 技术方案反复推翻 | PRD撰写 | 技术对齐太晚 | PRD前做技术预沟通 |
| 串通流于形式 | 全流程串通 | 只走正常流程 | 多角色多路径串通，强制测试边界 |

> 从0到1的产品设计，不怕卡点，怕的是不知道卡点在哪。提前识别、提前预防，比事后救火高效十倍。

本章小结：从0到1的产品设计，核心不是"做多少功能"，而是"想清楚做什么"。假设验证法帮你判断做不做，MVP帮你定义做什么，全流程串通帮你确保没遗漏。

| 本章核心知识点 | 一句话总结 |
|-------------|-----------|
| 假设验证法 | 想法→假设→验证→决策 |
| 假设验证常见错误 | 验证偏差、幸存者偏差、确认偏差——验证本身也可能骗你 |
| 最小化验证实验 | 最小成本+可信结论，预设否定标准 |
| 需求筛选 | 价值×可行×紧迫 |
| MVP定义 | 验证核心假设的最小功能集合 |
| MVP三种形态 | 冒烟测试验证需求、Wizard of Oz验证体验、功能精简验证产品 |
| 电梯演讲 | 30秒讲清产品思路 |
| 原型自检 | 主流程+交互+异常+数据+状态 |
| 全流程串通 | 模拟用户走一遍，发现漏洞 |
| 全流程checklist | 每个环节有交付物和验收标准 |
| 流程卡点识别 | 提前识别高危环节，预防胜于救火 |
